# 用 AI 写出你的 Resume Summary

> 这是「用 AI 构建你的简历」系列课程的第五节。我们终于要开始动手了——用 AI 帮你写出 3-4 个针对不同方向的 Summary，然后和你的简历主体内容进行排列组合。

## 课程导读

恭喜你走到这一步！

在前几节课里，我们讲了招聘流程、ATS 机制、应届生简历怎么写、有经验者简历怎么写。你已经理解了 Summary 是什么、为什么重要、好的 Summary 长什么样。

现在，是时候动手了。

这节课只有一个目标：**用 AI 帮你写出可以直接用在简历上的 Summary**。

不是理论，不是概念，而是实打实地产出几个 Summary 版本——针对你可能申请的不同岗位方向（比如 Backend、Full-Stack、Data 等），每个方向一个 Summary。写完之后，你就可以根据具体投递的岗位，选择最合适的那个版本，和你简历的主体内容搭配起来。

这就是 Tailoring 的核心思路：简历主体内容不用大改，但 Summary 要针对不同方向做调整。

---

## 学习目标

学完这节课，你会掌握：

1. **使用 `/summary-writer` 命令**——这是一个交互式的 AI 助手，帮你头脑风暴、梳理定位、写出初稿
2. **使用 `/summary-reviewer` 命令**——这是一个严苛的面试官角色，评估你的 Summary 能不能让人有动力继续看你的简历
3. **两个窗口协作的工作流**——一个写、一个审，来回迭代，直到产出一个真正好用的 Summary

---

## 前置准备：把你的背景信息放进 profile/

在开始之前，你需要让 AI 能够"读懂"你的背景。

**这一步非常重要。** 把你的基本信息放到项目的 `profile/` 目录下，而且要**尽可能展开写**。

不要只放一份粗糙的简历草稿。要把你的经历拆开，一条一条写清楚：

- 每个项目单独一条，写清楚你做了什么、用了什么技术、结果是什么
- 每段工作经历展开，不只是职位名称，而是你具体负责的事情
- 你引以为豪的成就，哪怕只是小的优化，也写下来
- 你的技术栈，分类列出来
- 任何你觉得"可能值得一提"的点

为什么要这么详细？

因为 AI 只能基于你给的信息来工作。你写得越详细，AI 就越能帮你挖掘亮点、找到定位。如果你只给一个干巴巴的简历，AI 也只能输出干巴巴的 Summary。

把 profile/ 当作你的"素材库"。你不需要在这里写得很漂亮，这不是给面试官看的，这是给 AI 看的背景信息。越详细越好，越具体越好。

如果 `profile/` 目录是空的，AI 会直接问你要信息——但提前准备好会让整个过程更顺畅，效果也更好。

## 第一步：用 summary-writer 写初稿

### 这个工具是什么

`/summary-writer` 是一个交互式的 AI 助手。它的工作流程是这样的：

1. **读取你的 profile**——理解你的背景
2. **分析你的方向**——根据你的经历，提出 2-4 个你可以定位的职业方向（比如 Backend Engineer、ML Engineer、Data Engineer）
3. **引导你思考**——问你一些关键问题，帮你想清楚：你想怎么定位自己？你最想突出什么？你要投什么类型的公司？
4. **写出初稿**——根据讨论的结果，写出一个 Summary 初稿
5. **保存结果**——把 Summary 存到 `resume/summary/` 目录下，文件名对应你的方向（比如 `backend.md`、`fullstack.md`）

### 怎么用

打开 Terminal，启动 Claude Code，然后输入：

```
/summary-writer
```

就这么简单。接下来就是交互式的对话。

AI 会先读你的 profile，然后开始问你问题。回答这些问题的过程，其实就是帮你理清思路的过程。

比如它可能会问：
- "我看到你做过后端和数据相关的工作，你想往哪个方向发展？"
- "你在 XX 项目里的具体贡献是什么？有没有可量化的成果？"
- "你想投大厂还是创业公司？新人还是资深？"

回答这些问题之后，它会给你写一个 Summary 初稿。

### 关于长度

对于应届生来说，Summary 一般就是两行。

AI 可能会写得稍微长一点——这没关系。写长了你可以删，删比从零开始写容易多了。

你可以：
- 把一些形容词删掉
- 把不那么重要的信息删掉
- 把一些表达压缩得更紧凑

AI 的作用是帮你理清思路，给你一个"可以马上用、稍微改改就能上"的初稿。最终的长度和措辞，你自己决定。

### 输出位置

Summary 会被保存到：

```
resume/summary/{方向}.md
```

比如你写了一个后端方向的 Summary，它会保存为 `resume/summary/backend.md`。

这样你就可以积累多个版本，之后投不同岗位的时候直接调用。

## 第二步：用 summary-reviewer 评估和优化

### 这个工具是什么

`/summary-reviewer` 是一个模拟严苛面试官的 AI 角色。

它的设定是：一个有 10 年经验的 Hiring Manager，每个月看几百份简历，见多识广，眼光很毒。它会站在面试官的角度评估你的 Summary：

**如果我是这个岗位的 Hiring Manager，看到这个 Summary，我会不会有兴趣继续看这份简历？还是直接 Pass 掉？**

它会给你打分、指出问题、给出修改建议。

### 需要准备什么

你需要给 AI 三个信息：

1. **Job Description**——你的目标岗位描述。可以放在 `application/job-description.md` 文件里，或者直接在对话中告诉 AI
2. **你的简历**——让 AI 了解你的整体背景
3. **你的 Summary**——这是它要评估的对象。如果你的简历里已经包含了 Summary，就不需要重复给

### 怎么用

打开 Terminal，启动 Claude Code，然后输入：

```
/summary-reviewer
```

AI 会先检查有没有现成的文件：
- 检查 `application/job-description.md` 里有没有 JD
- 检查 `resume/summary/` 目录下有没有你之前写的 Summary

如果都有，它会问你要评估哪个 Summary。如果缺少某些信息，它会让你提供。

### 评估维度

它会从以下几个维度给你评分：

- **Position Fit**：Summary 和 JD 是否匹配？
- **Credibility**：说的话可信吗？有没有夸大？
- **Differentiation**：和其他候选人比，有没有独特之处？
- **Scannability**：3 秒钟能不能抓住关键信息？
- **Language Quality**：语言是否专业、流畅？

最后会给出总分、总体评价、以及"如果让我来改，我会怎么写"的示例。

### 它的反馈风格

这个 AI 角色设定得比较严苛。它会直接告诉你哪里写得不好：

- "这个词太大了，你一个应届生用 'Architected'，我会怀疑你其他描述的真实性。"
- "这句话太空洞了，'Passionate about technology' 跟没说一样。"
- "JD 要的是 Data Engineer，你 80% 的内容在讲 Web 开发，方向不对。"

不要觉得被打击了——这就是它的价值。它帮你在投简历之前发现问题，比被真正的面试官 Pass 掉要好得多。

## 第三步：两个窗口协作，迭代优化

现在你学会了两个命令。它们最好的使用方式是**一起用**。

### 推荐工作流

1. **打开两个 Terminal 窗口**，每个窗口启动一个 Claude Code 实例
2. **左边窗口运行 `/summary-writer`**——用来写和修改 Summary
3. **右边窗口运行 `/summary-reviewer`**——用来评估和挑战

工作流程是这样的：

1. 在左边窗口写出一个 Summary 初稿
2. 把这个 Summary 扔给右边窗口评估
3. 右边窗口会给出反馈和修改建议
4. 把这些建议复制回左边窗口，让它改进
5. 改完再扔给右边窗口再评估
6. 重复，直到右边窗口给出满意的评分

一般来说，经过 2-3 轮迭代，你就能得到一个相当不错的 Summary。

### 为什么要两个窗口

因为这两个 AI 角色的立场不同。

Summary Writer 是你的教练，帮你挖掘亮点、组织语言、写出初稿。它偏向于"帮你写出来"。

Summary Reviewer 是一个挑剔的面试官，专门找问题、挑毛病、提要求。它偏向于"不够好就打回去"。

两个角色之间的张力，会推动你的 Summary 越来越好。

## 最终产出

经过这个流程，你会在 `resume/summary/` 目录下积累几个 Summary 文件：

```
resume/summary/
├── backend.md
├── fullstack.md
├── ml-engineer.md
└── data-engineer.md
```

每个文件对应一个方向。之后投简历的时候，根据岗位选择合适的 Summary，和你的简历主体拼在一起就行了。

这就是 Tailoring 的最简单形式：主体不用大改，只换 Summary。

## 课程作业

这节课的作业分两个部分。

### 第一部分：提交一个 Summary 给导师点评

1. 用 `/summary-writer` 写一个 Summary 初稿
2. 用 `/summary-reviewer` 评估这个 Summary
3. 根据反馈修改，然后再评估
4. **来回至少 3 轮**，直到你觉得差不多了

完成后，在课程卡片的评论区提交以下内容：

- 你的 Summary（最终版本）
- 你的简历（或 profile 内容）
- 你参考的 Job Description
- `/summary-reviewer` 的最后一轮点评

导师会看你的提交，确认你已经掌握了这个工作流，并给你进一步的修改建议。

### 第二部分：准备 3 个不同侧重点的 Summary

在导师确认你会用了之后，自己回去继续练习。

目标是准备 **3 个不同方向** 的 Summary。人设可以一样，但定位和侧重点不同。比如：

- 一个定位去创业公司，强调 ownership 和 ship fast
- 一个定位做产品相关的工程师，强调 user-facing 和 impact
- 一个定位 AI 原生应用方向，强调 LLM、prompt engineering、RAG
- 一个定位 data-driven 岗位，强调数据处理、分析、pipeline

不需要都是这四个方向，根据你自己的背景和求职目标来定。

关键是：**同一个人，可以有不同的定位方式**。这 3 个 Summary 会成为你之后投简历时的候选项。

## 关键洞察

让我总结一下这节课的核心要点。

**第一，AI 是工具，不是替代品。** 它帮你理清思路、加速写作、提供反馈——但最终的判断和决策还是你自己做。它给的初稿你要改，它给的建议你要筛选。

**第二，写长了比写短了好处理。** 让 AI 多写一点，你再删减。删比从零开始写容易。

**第三，迭代是正常的。** 没有人一次就能写出完美的 Summary。写-评-改-再评，这是正常的流程。

**第四，准备多个版本。** 不同岗位需要不同的定位。花时间为每个方向准备一个 Summary，投递时就可以灵活组合。

**第五，最终验证要自己做。** AI 评估完之后，还是把 Summary 贴到你的简历文档里，看看实际长度是不是两行。超了就删，短了就加。

## 导师寄语

很多人把写 Summary 想得太复杂了——觉得要一次写到完美，觉得 AI 会直接给出终极答案。

不是这样的。

Summary 本质上是一个定位问题：你要告诉读者你是谁、你能做什么、为什么值得他们继续看。这个定位不是凭空产生的，是通过不断思考、尝试、反馈、修改而逐渐清晰的。

AI 加速了这个过程。以前你可能要自己想、自己改、找朋友帮忙看。现在你可以和 AI 对话，让它问你问题、帮你提炼、给你反馈。

但最终，Summary 说的是你的故事。你要对它负责。

不要期待 AI 给你一个完美的答案——期待它给你一个好的起点，然后你自己打磨到满意为止。

祝你写出一个让人想继续看下去的 Summary。
