# 应届生简历怎么写：没有工作经验，如何让人愿意面试你？

> 这是「用 AI 构建你的简历」系列课程的第三节。前两节我们讲了招聘流程和筛选机制，这一节进入实操：应届生简历到底怎么写？

## 课程导读

欢迎来到第三节。

前两节我们做了很多铺垫。你现在知道了简历会给谁看——Recruiter、Hiring Manager，以及不同场景下的不同读者。你也知道了 ATS 是怎么回事——它是个数据库，不是裁判，真正筛你的是真人。

现在，终于到了实操的部分：怎么写一份能拿到面试的简历？

这一节专门针对应届生。如果你是还没毕业或刚毕业不久、没有全职工作经验的人，这节课就是为你准备的。

应届生写简历有一个核心困境：你没有工作经验，但简历就是要展示你能做什么。怎么办？

在开始之前，有一点很重要：这是一篇中文教程，但我们教的是如何写**北美求职的英文简历**。所以后面所有简历上的描述、Bullet Points、Summary 示例，都会是英文的。这是你实际要写在简历上的内容。

---

## 学习目标

先回顾一下第一节讲过的内容。应届生找工作，主要面对两种场景。

第一种是大公司校园招聘。Google、Meta、Amazon 这些公司有专门的 University Recruiting 团队，会去学校办 Career Fair、Info Session，批量收简历、批量筛选、批量面试、批量发 Offer。这种模式下，你的简历会和成千上万份简历放在一起，经过标准化筛选。硬指标权重很高：学校、GPA、专业、关键词匹配。

第二种是中小公司岗位招聘。初创公司、中型科技公司，它们没有专门的校招流程。Hiring Manager 缺人了，发个 JD，简历就开始涌进来。问题是——你的简历会和有 2-3 年经验的人放在同一个池子里。这是很多应届生没有意识到的：你不是在和其他应届生竞争，你是在和职场老兵竞争。

这两种场景对简历的要求不太一样，但核心问题是相同的：没有工作经验，怎么让人觉得你值得面试？

学完这节课，你会知道应届生简历应该有什么结构，要不要写 Summary，项目经历怎么体现"我做了什么"，以及最常见的错误是什么。

---

## 一个关键认知：你可能需要不止一份简历

在讲简历结构之前，我想先说一个很多应届生没有意识到的事情。

你可能需要不止一份简历。

为什么？因为即使是同一个岗位名称，实际工作内容可能完全不同。

拿 "AI Developer" 这个头衔来说。不同公司招的 AI Developer 可能侧重点完全不同：有的公司需要偏 SDE 方向的，主要写 Production Code、做系统集成；有的需要偏 AI/ML 方向的，主要做模型训练和优化；有的需要偏 Data 方向的，主要做数据处理和 Pipeline。

如果你用同一份简历去申请这三种岗位，你的简历对每一种都不够聚焦。

更进一步，如果你的技能允许你申请多个不同的 Job Family——比如 Software Engineer、Data Engineer、Machine Learning Engineer——那你应该准备多个版本的简历，每个版本针对一个方向优化。

这意味着什么？你的 Summary 不同，你突出的技能不同，你强调的项目不同，甚至你 Bullet Points 的措辞都不同。

我们后面会详细讲怎么做这种 Tailoring。现在先记住这个核心认知：一份简历打天下的时代过去了。

## 应届生简历的核心结构

让我们先搞清楚一份应届生简历应该包含哪些模块，以及它们的顺序。

对于应届生来说，推荐的顺序是这样的：

1. Contact Information（联系方式）
2. Summary（个人简介）— 如果你有多个方向，这个放最上面
3. Education（教育背景）
4. Skills（技能）
5. Projects（项目经历）
6. Work Experience（实习经历，如果有的话）

注意这个顺序。如果你有明确的求职方向、需要用 Summary 来定位自己，Summary 应该放在最上面，比 Education 还高。因为它是告诉读者"我是谁、我能做什么"的第一句话。

下面我们一个一个来讲。

---

## Contact Information（联系方式）

这部分看起来最简单，但很多人偏偏在这里出问题。

你需要包含：姓名、Email、电话号码、所在城市。如果你有 LinkedIn 和 GitHub，也可以放上去——但前提是它们看起来像样。

关于 GitHub，这是一个很容易减分的地方。永远不要链接一个空荡荡的 GitHub。如果你的 GitHub 上只有几个空 Repository，或者最后一次 Commit 是两年前，那就别放了。一个挂掉的 Heroku Deployment、一个写了一半的项目，这些只会减分。要么整理好再放，要么就不放。

还有一点很多人忽略：保持所有信息视觉上的一致性。URL 没对齐、行间距不统一、LinkedIn 链接本可以放在同一行却另起一行——这些小细节会让一个自称"注重细节"的人显得自相矛盾。

---

## Summary（个人简介）：什么时候需要，怎么写

这是一个经常被问到的问题：应届生简历要不要在开头写一个 Summary？

我的建议是：**如果你有明确的求职方向，并且需要告诉读者"我是谁"，就写。如果你不打算针对每个申请定制 Summary，那就不写。**

一个不痛不痒的 Summary 只会浪费你宝贵的简历空间。比如这种：

> *Passionate developer seeking challenging opportunities to grow and contribute to innovative teams.*

这种句子毫无信息量。每个人都可以写这句话。它没有告诉读者任何关于你的具体信息。

但如果你有明确的方向，Summary 可以非常有效。它可以在几秒钟内告诉读者：你是谁，你能做什么，你适合什么岗位。

### Summary 应该放在什么位置？

如果你写 Summary，它应该放在简历最上面，比 Education 还高。

为什么？因为 Summary 的作用是"定位"。在 Recruiter 开始扫读你的简历之前，Summary 先告诉他们：这个人是 Front-End Engineer、是 Data Scientist、还是 Machine Learning Engineer。这个定位会影响他们如何理解后面所有的内容。

### 多个方向 = 多个 Summary = 多份简历

这里就回到了我们前面说的：如果你想申请多个方向，你需要准备多份简历，每份有不同的 Summary。

让我用几个例子来说明。假设你是一个 CS 应届生，做过一些 Web 开发、一些数据分析、一些机器学习项目。你想申请 SDE、Data Engineer、Machine Learning Engineer 三个方向。

**针对 Software Engineer (SDE) 岗位的 Summary：**

> *Computer Science new grad with hands-on experience building full-stack web applications. Proficient in Python, JavaScript, React, and Node.js. Developed and deployed 3 production-ready projects with 500+ users.*

这个 Summary 强调的是：Full-Stack 开发能力、Web 技术栈、有 Production 经验。

**针对 Data Engineer 岗位的 Summary：**

> *Computer Science new grad with strong foundation in data systems and ETL pipelines. Experienced with Python, SQL, Spark, and AWS. Built automated data pipelines processing 100K+ records daily in academic projects.*

这个 Summary 强调的是：数据系统、Pipeline 构建、大数据处理能力。

**针对 Machine Learning Engineer 岗位的 Summary：**

> *Computer Science new grad specializing in machine learning and deep learning. Proficient in Python, PyTorch, and TensorFlow. Implemented and trained CNN models achieving 94% accuracy on image classification tasks.*

这个 Summary 强调的是：ML/DL 能力、模型训练经验、有具体的 Metrics。

看出区别了吗？同一个人，三个完全不同的定位。

### 更多岗位方向的 Summary 示例

让我再给你一些不同岗位方向的 Summary 示例，帮你理解怎么针对不同方向做定位。

**AI Developer（偏应用开发）：**

> *Computer Science new grad focused on building AI-powered applications. Experienced with Python, LangChain, and OpenAI APIs. Developed a RAG-based chatbot handling 1000+ queries in production environment.*

**AI Engineer（偏模型和系统）：**

> *Computer Science new grad with expertise in deploying ML models at scale. Proficient in Python, PyTorch, Docker, and Kubernetes. Built end-to-end ML pipelines from training to inference serving.*

**Data Analyst：**

> *Statistics and Computer Science new grad with strong analytical skills. Proficient in Python, SQL, Tableau, and Excel. Conducted data analysis projects uncovering insights that informed business decisions.*

**Data Scientist：**

> *Computer Science new grad with a focus on statistical modeling and machine learning. Experienced with Python, R, scikit-learn, and Pandas. Built predictive models with 85%+ accuracy on real-world datasets.*

**Cloud Engineer：**

> *Computer Science new grad with hands-on experience in cloud infrastructure. Proficient in AWS (EC2, S3, Lambda), Terraform, and Docker. Deployed and managed cloud-based applications with 99.9% uptime.*

**Solution Architect（应届生较少见，但有些公司招）：**

> *Computer Science new grad with broad technical foundation across cloud, data, and application development. AWS Certified. Designed system architectures for 3 academic capstone projects involving microservices and distributed systems.*

**Financial Engineer / Quantitative Developer（金融科技方向）：**

> *Mathematics and Computer Science new grad with strong foundation in quantitative analysis. Proficient in Python, C++, and statistical modeling. Developed trading strategy backtesting framework processing 1M+ historical data points.*

你会注意到，每个 Summary 都有相同的结构：

1. 你是谁（专业背景）
2. 你的核心技能是什么
3. 你做过什么（有具体的成果）

但强调的技能和经验完全不同。

## Education（教育背景）

对于应届生来说，Education 是你最有说服力的资质之一。如果你没有写 Summary，Education 就放在最上面；如果有 Summary，Education 紧随其后。

Education 部分应该包含什么？学校名称、学位、专业、毕业时间（或预计毕业时间）。如果 GPA 亮眼（比如 3.5 以上），写上。如果不高，就省略——没人会追问你为什么没写。

你可以列出 Relevant Coursework，尤其是和目标岗位相关的课程。但要针对你申请的方向来选择列出哪些课程。

申请 Machine Learning Engineer？列出：

> *Relevant Coursework: Machine Learning, Deep Learning, Statistical Learning, Linear Algebra, Probability and Statistics*

申请 Software Engineer？列出：

> *Relevant Coursework: Data Structures and Algorithms, Operating Systems, Database Systems, Software Engineering, Computer Networks*

申请 Data Engineer？列出：

> *Relevant Coursework: Database Systems, Distributed Systems, Data Mining, Cloud Computing, Big Data Analytics*

还有一点：如果你在学校有值得一提的成就——比如在 Hackathon 获奖、领导编程社团、担任课程 TA——也可以写在 Education 部分。这些能展示你不只是一个被动上课的学生。

## Skills（技能）

Skills 部分要精准、有针对性。

这里最常见的错误是：把你接触过的每一项技术都列上去。Python、Java、C++、JavaScript、TypeScript、React、Vue、Angular、Node.js、Django、Flask、Spring、TensorFlow、PyTorch、AWS、GCP、Azure……

如果你列了 15 种编程语言和 20 个 Framework，它传递的信号不是"这个人很厉害"，而是"这个人可能哪种都不精通"。

Hiring Manager 要的是 Specialist，不是 Generalist。

Skills 部分也需要根据你申请的方向来调整。同一个人，针对不同岗位，Skills 部分可以完全不同。

**SDE 岗位的 Skills：**

> *Languages: Python, JavaScript, TypeScript, Java*
> *Frameworks: React, Node.js, Express, Django*
> *Tools: Git, Docker, PostgreSQL, Redis*

**Data Engineer 岗位的 Skills：**

> *Languages: Python, SQL, Scala*
> *Data Tools: Spark, Airflow, dbt, Kafka*
> *Cloud: AWS (S3, Redshift, Glue), Snowflake*

**Machine Learning Engineer 岗位的 Skills：**

> *Languages: Python, C++*
> *ML Frameworks: PyTorch, TensorFlow, scikit-learn*
> *Tools: Docker, MLflow, Weights & Biases, Jupyter*

**Cloud Engineer 岗位的 Skills：**

> *Cloud Platforms: AWS (EC2, S3, Lambda, RDS), GCP*
> *Infrastructure: Terraform, Docker, Kubernetes*
> *Languages: Python, Bash, YAML*

看到规律了吗？每个版本都只列出和目标岗位最相关的技能，而且按照逻辑分组。

一个实用的检验方法：只列出你在面试中能自信谈论的内容。如果面试官问你"你用 Kubernetes 做过什么"，你能回答吗？如果不能，就不要列。

## Projects（项目经历）：应届生简历的核心

这是应届生简历最重要的部分。

没有工作经验，你的项目就是你的经验。Projects 部分证明你不只是上课、做作业，你在课堂之外也在写代码、解决真实问题。

### Personal Project vs Homework

这里有一个关键区别：**Personal Project 和 Homework 不是一回事。**

如果你班上所有人都写了同一个 Blackjack CLI 游戏，那不是 Personal Project，那是 Homework。把它列出来不仅不加分，反而暴露了你缺乏独立主动性。这是一个 Red Flag。

雇主在寻找的是什么？那些出于真正好奇心、在课堂和工作之外主动 Build Things 的人。热爱工作的人会做得更多，也会进步得更快。

什么样的项目值得列？一个解决真实问题的项目，一个展示技术深度的项目，一个探索有意义的技术问题的项目。

### 项目描述怎么写：Action + Technical Detail + Impact

很多应届生的项目描述是这样的：

> *Built a To-Do List app using React and Node.js.*

这句话有什么问题？它只说了你用了什么技术，没有说你做了什么、解决了什么问题、产生了什么结果。

一个好的项目描述应该遵循这个公式：**Action + Technical Detail + Impact**。

**Action（做了什么）**

用具体的动词开头，描述你做了什么。

但这里有一个重要提醒：**应届生要避免使用一些"大词"**。

什么是大词？比如 "Spearheaded"、"Architected"、"Pioneered"、"Orchestrated"。这些词暗示的是领导角色、架构师级别的工作。如果你是一个应届生，独立做一个课程项目，用这些词会显得不真实。

什么词适合应届生用？

- Built — 构建了
- Developed — 开发了
- Implemented — 实现了
- Designed — 设计了
- Created — 创建了
- Optimized — 优化了
- Automated — 自动化了
- Integrated — 集成了

这些词足够有力，又不会让人觉得你在夸大。

**Technical Detail（技术细节）**

说明你用了什么技术、怎么实现的。但不要只是堆砌技术名词，要让人看出你真正用这些技术做了什么。

不好的写法：

> *Used React, Node.js, MongoDB, Express.*

好的写法：

> *Built responsive frontend with React and TypeScript, RESTful API with Node.js/Express, and MongoDB for data persistence.*

第二种写法让人看出你理解每个技术在系统中的角色。

**Impact（影响/结果）**

这是最容易被忽略的部分，也是需要最谨慎的部分。

关于写 Impact 的数字，有一个重要原则：**如果你不能经得起追问，就不要写。**

什么意思？如果你写"将页面加载时间降低了 50%"，面试官可能会问：

- 你怎么测量的？
- 之前是多少，之后是多少？
- 你用什么工具测量？
- 你做了什么优化导致了这个改进？

如果你回答不上来，这个数字就会变成一个 Red Flag，让人怀疑你简历上其他内容的真实性。

那怎么办？在写 Impact 之前，先做好功课。你可以问自己这些问题：

- 这个数字在业内的合理范围是多少？
- 我是怎么衡量的？用的什么方法、什么工具？
- 如果被追问，我能解释清楚吗？

你甚至可以在写简历之前，和 AI 讨论一下你的 Impact 数字是否合理。比如：

> "我做了一个 Web 应用，优化了图片加载，页面加载时间从 3 秒降到 1.5 秒。这个数字在业内是否合理？我用 Chrome DevTools 的 Lighthouse 测量的，这个方法对吗？"

把思想实验做清楚，再写到简历上。

如果实在没有可靠的数字，你可以描述项目解决了什么问题，或者它的潜在应用场景。没有数字的 Impact 也是 Impact。

> *A Chrome extension that helps fitness enthusiasts match Spotify songs by BPM for workout playlists, eliminating manual playlist curation.*

这句话没有数字，但让人看出：这个项目解决了一个真实问题。

### 不同方向的项目描述示例

让我给你一些不同岗位方向的项目描述示例，帮你理解怎么针对不同方向写 Bullet Points。

**SDE / Full-Stack 方向：**

项目：一个 Task Management Web App

> *Developed a full-stack task management application with user authentication, real-time updates, and team collaboration features.*
>
> *Built frontend with React and TypeScript, implemented RESTful API with Node.js/Express, used PostgreSQL for data storage and Redis for session management.*
>
> *Deployed on AWS EC2 with Docker, set up CI/CD pipeline using GitHub Actions. Currently serves 200+ registered users.*

**Data Engineer 方向：**

项目：一个数据 Pipeline 项目

> *Built an automated ETL pipeline to collect, transform, and load social media data for sentiment analysis.*
>
> *Used Python and Apache Airflow for orchestration, processed 50K+ tweets daily, stored results in PostgreSQL and visualized trends in Grafana.*
>
> *Reduced manual data collection time from 4 hours to 15 minutes through automation.*

**Machine Learning Engineer 方向：**

项目：一个图像分类模型

> *Implemented a CNN-based image classification model to identify plant diseases from leaf images.*
>
> *Built data pipeline with Python and OpenCV, trained model using PyTorch on 10K+ labeled images, achieved 92% validation accuracy.*
>
> *Deployed model as a REST API using FastAPI and Docker, enabling real-time inference.*

**Data Scientist 方向：**

项目：一个预测模型

> *Developed a predictive model to forecast customer churn for a simulated e-commerce dataset.*
>
> *Performed exploratory data analysis and feature engineering using Pandas and scikit-learn, built and compared Random Forest, XGBoost, and Logistic Regression models.*
>
> *Final model achieved 87% AUC-ROC, identified top 5 features driving customer churn.*

**AI Developer 方向：**

项目：一个 RAG Chatbot

> *Built a RAG-based chatbot that answers questions about university course catalogs using natural language.*
>
> *Implemented document ingestion with LangChain, vector storage with Pinecone, and query interface with OpenAI GPT-4 API.*
>
> *Handles 100+ queries daily with 85% user satisfaction rate based on feedback collection.*

**Cloud Engineer 方向：**

项目：一个云部署项目

> *Designed and deployed a microservices-based web application on AWS using infrastructure as code.*
>
> *Used Terraform to provision EC2, RDS, and S3 resources, Docker for containerization, and ECS for orchestration.*
>
> *Achieved 99.5% uptime over 3-month monitoring period with automated scaling and health checks.*

**Financial Engineering / Quant 方向：**

项目：一个量化策略回测项目

> *Developed a backtesting framework for evaluating momentum-based trading strategies on historical stock data.*
>
> *Built data pipeline ingesting 5 years of daily price data from Yahoo Finance API, implemented strategy logic in Python with NumPy and Pandas.*
>
> *Backtested strategy achieved 12% annualized return with Sharpe ratio of 1.3 on S&P 500 constituents.*

## Work Experience（实习经历）

如果你有实习经历，这部分非常有价值。

即使只是一个暑期实习，它也是"有人付钱让你做事"的证明。这和课堂项目有本质区别。

实习经历的写法和项目类似，但有一些额外要点。

### 建立 Ownership

在描述中展示这是"你的"工作，不只是"你参与的"工作。

不好的写法：

> *Participated in the team's data analysis project.*

好的写法：

> *Owned the data cleaning and preprocessing module, processing 100K+ customer records daily.*

"Owned"、"Led"、"Responsible for"这些词传递的是你有 Ownership，不是一个只会执行指令的实习生。

但再次提醒：要真实。如果你只是参与了一小部分，不要写得像你主导了整个项目。

### 量化 Impact（但要谨慎）

实习的好处是你可能有真实的业务数据。但同样的原则适用：**如果不能经得起追问，就不要写。**

一些合理的 Impact 描述：

> *Reduced report generation time from 2 hours to 15 minutes through automation.*
>
> *Improved model inference speed by 30% through code optimization and caching.*
>
> *Contributed to a feature that was shipped to 10K+ users.*

注意这些描述都是可验证的：你知道之前要 2 小时，你知道你优化后是 15 分钟，你有方法解释你怎么测量的。

### 关于数据保密

这里有一个提醒：始终假设内部数据是机密的。在简历上使用具体 Metrics 之前，最好征求许可，或者用百分比、倍数来模糊化，而不是使用原始数字。

比如，你不能说"公司的 DAU 从 100 万增长到 150 万"，但你可以说"DAU increased by 50%"。

## 应届生简历的常见错误

让我讲一些我见过最多的应届生简历错误。

### 错误一：列太多技术

你列了 15 种编程语言和 20 个 Framework，传递的信号是你可能哪种都不精通。聚焦在你真正擅长的技术上。针对你申请的方向，列出最相关的 3-5 项核心技能。

### 错误二：把课堂作业当 Personal Project

如果你班上所有人都做了同一个项目，那就是 Homework，不是 Personal Project。你需要有至少一两个真正独立完成、出于个人兴趣或解决真实问题的项目。

### 错误三：只列技术、不说做了什么

> *Built an app using React, Node.js, MongoDB.*

这句话没有告诉读者你做了什么。你解决了什么问题？有什么功能？产生了什么结果？

### 错误四：用无法证实的夸张数字

> *Improved performance by 500%.*

如果面试官问你怎么测量的、之前是多少、之后是多少、你做了什么导致这个改进，你能回答吗？如果不能，不要写。

### 错误五：使用应届生不该用的"大词"

"Spearheaded"、"Architected"、"Pioneered"——这些词暗示的是领导角色、架构师级别的工作。如果你只是一个独立完成课程项目的学生，用这些词会显得不真实。用 "Built"、"Developed"、"Implemented" 这些词足够了。

### 错误六：GitHub 链接指向空 Repository

放了 GitHub 链接，点进去一看——空的，或者最后一次 Commit 是两年前。要么整理好再放，要么就不放。

### 错误七：简历超过一页

如果你是应届生，你不需要两页简历。如果你觉得一页装不下，说明你还不够精炼。Recruiter 在一份简历上平均只花大约六秒钟。如果他们需要翻页，你基本已经出局了。

### 错误八：一份简历投所有岗位

你用同一份简历投 SDE、Data Engineer、ML Engineer，每个方向都不够聚焦。针对不同方向准备不同版本，会显著提升你的面试转化率。

## 两种场景下的策略差异

回到开头说的两种场景。应届生面对大公司校招和中小公司直投，策略上有什么区别？

### 大公司校招

大公司校招是批量筛选。你的简历会和成千上万份放在一起，经过 Recruiter 快速扫读。

在这种场景下，你需要确保通过标准化筛选。学校、GPA、关键词匹配——这些硬指标很重要。确保你的简历能让 Recruiter 快速打勾。你需要展示"符合标准"。大公司校招看的是你是否"合格"，而不是你有多独特。确保基本要素都有：相关专业、相关技能、相关项目。技术栈匹配也很重要。如果 JD 要求 Java 和 Spring，确保你的 Skills 和 Projects 里有这些。

### 中小公司直投

中小公司没有专门的校招流程，你的简历会和有经验的人放在同一个池子里。

在这种场景下，你需要脱颖而出。你在和工作了 2-3 年的人竞争。"符合标准"不够，你需要让 Hiring Manager 觉得"这个人值得一试"。你需要展示潜力。你没有工作经验，但你可以展示学习能力、主动性、解决问题的能力。你的 Personal Projects 在这里特别重要。你需要讲一个故事。为什么你虽然没有经验，但值得被录用？你的项目、你的技术深度、你解决问题的方式——这些能讲出一个让人信服的故事。

两种场景的共同点是：清晰度、Impact、可扫读性依然是核心。只是侧重点有所不同。

## 核心洞察

让我总结一下这节课的核心要点。

第一，你可能需要不止一份简历。针对不同岗位方向，准备不同版本，每个版本有不同的 Summary、不同的 Skills 强调、不同的项目顺序。

第二，Summary 如果写，就放在最上面。它的作用是定位你是谁、你能做什么。

第三，Projects 是应届生简历最重要的部分。它证明你不只是上课，你在课堂之外也在解决真实问题。

第四，项目描述要遵循 Action + Technical Detail + Impact 的公式。用真实的、可验证的动词和数字，不要用大词，不要写无法证实的夸张数字。

第五，关于 Impact 的数字：如果不能经得起追问，就不要写。写之前先想清楚怎么测量、怎么解释。

第六，Skills 要精准，针对你申请的方向来调整。不要列一长串所有接触过的技术。

## 导师寄语

我见过很多应届生，技术能力其实很强，但简历看起来像是花了 15 分钟草草拼凑的。

他们列了一堆技术名词，但没有展示自己做过什么。他们把所有课堂项目都列上去，但没有一个真正独立完成的作品。他们的 Impact 数字看起来很漂亮，但经不起追问。

这不是能力问题，是表达问题。

你的简历是在回答一个问题：为什么我们应该面试这个人？

对于应届生来说，答案不是"因为我学过很多技术"。答案是"因为我能做事"。

怎么证明你能做事？用你做过的事来证明。

这就是为什么 Personal Projects 如此重要。它们是你唯一能展示"我不只是学了，我做了"的地方。

如果你觉得自己目前的项目还不足以撑起一份有力的简历，那就是一个信号：去 Build More。

找一个你真正感兴趣的问题，用你学过的技术去解决它。把它做完、部署上线、让真人用起来。这个过程本身就是最好的学习，而你会得到一个能写在简历上的真实成果。

在写 Impact 数字之前，先想清楚你怎么测量、你能怎么解释。可以和 AI 讨论，可以自己写代码做实验，可以查行业基准。把功课做在前面，简历上的每一个字都经得起追问。

热爱工作的人会做得更多，也会进步得更快。让你的简历展示这一点。
