# Resume Summary Writing Guide

## What is a Summary?

A Summary (also called Professional Summary or Profile) is a short paragraph at the top of your resume that:
- Positions who you are
- Showcases your core competencies
- Indicates your career direction

It's the first thing readers see and shapes how they interpret everything that follows.

## Why Different Summaries for Different Directions?

The same experience can be framed from multiple angles.

**Example: A candidate who built an AI Agent project**

This project involved:
- Vertical industry knowledge (healthcare prior authorization)
- SQL + industry data processing
- Agent development (LangChain, RAG)
- AWS Cloud deployment
- Full software engineering lifecycle

For different positions, the Summary can be completely different:

**SDE-focused Summary:**
> Full-Stack Engineer with hands-on experience building production-grade applications. Proficient in Python, TypeScript, React, and AWS. Developed and deployed an AI-powered enterprise system serving healthcare clients.

Emphasis: Software engineering capabilities, Full-Stack, Production experience

**AI Developer-focused Summary:**
> AI Developer specializing in building LLM-powered applications for enterprise use cases. Experienced with LangChain, RAG architectures, and prompt engineering. Built an AI agent system automating healthcare workflows with 85% accuracy.

Emphasis: AI/LLM capabilities, Agent development, Vertical applications

**Data Engineer-focused Summary:**
> Data Engineer with experience building data pipelines for healthcare analytics. Proficient in Python, SQL, and AWS data services. Designed ETL workflows processing medical records and insurance data for AI-driven decision support.

Emphasis: Data processing, Pipeline construction, Industry data

Same project, three completely different narrative angles.

## Summary Structure

### Length
- Should NOT exceed 2 lines when displayed on a standard resume
- Approximately 200-300 characters
- Readable in one quick scan
- After drafting, copy to Word or Google Docs to verify actual length
- You may need to trim adjectives or modifiers to fit

### Standard Structure
```
[Identity] with [years/background] [core area]. [Key technologies].
[Standout achievement + metrics] OR [Unique value proposition].
```

### Elements to Include

**Must have:**
- Professional identity (Software Engineer, Data Scientist, etc.)
- Core tech stack or capabilities
- At least one specific achievement or differentiator

**Optional:**
- Years of experience (required for experienced, optional for new grads)
- Target industry or company type
- Certifications (e.g., AWS Certified)

**Avoid:**
- "I am..." or "My goal is..."
- "Passionate", "Motivated", "Team player"
- Vague adjectives without substance

## Strength Dimensions

Different people have different core strengths. Your Summary should reflect these differences:

### Builder
Emphasis: shipping products, building features, production systems
> ...building user-facing products... shipped 5 features serving 500K+ users...

### Analyst
Emphasis: data-driven insights, experimentation, decision support
> ...driving data-informed decisions... A/B testing initiatives... uncovering actionable insights...

### Architect
Emphasis: system design, scalability, technical leadership
> ...designing scalable architectures... leading technical decisions... microservices migration...

### Communicator
Emphasis: cross-functional collaboration, stakeholder management
> ...collaborating with product and business teams... translating technical concepts... leading initiatives across 4 teams...

### Domain Expert
Emphasis: industry-specific knowledge, vertical expertise
> ...specializing in healthcare technology... financial systems expertise... deep understanding of regulatory requirements...

## Common Mistakes

### 1. Too Generic
**Bad:** Passionate software engineer seeking challenging opportunities to grow and contribute.
**Why bad:** Every candidate says this. Zero information content.

### 2. Technology Dumping
**Bad:** Proficient in Python, Java, Go, JavaScript, TypeScript, C++, Ruby, React, Vue, Angular, Node, Django, Flask, Spring...
**Why bad:** Lists don't prove expertise. Shows breadth but no depth.

### 3. Vague Claims
**Bad:** Improved system performance and enhanced user experience.
**Why bad:** "Improved" and "enhanced" mean nothing without numbers.

### 4. Using Big Words as New Grad
**Bad (for new grad):** Spearheaded architectural transformation of enterprise systems.
**Why bad:** New grads don't "spearhead" anything. Sounds fake.

### 5. First Person
**Bad:** I am a software engineer who loves building products.
**Why bad:** Resumes use third-person implied. No "I" or "my".

## The Acid Test

Before finalizing a summary, ask:
1. Does this tell the reader WHO I am in 3 seconds?
2. Is there at least ONE specific, concrete detail?
3. Could anyone else write this exact same sentence? (If yes, it's too generic)
4. Would I feel confident if asked about any claim here?
5. Does it fit within 2 lines on a standard resume format? (Paste into Word/Google Docs to check)

## Tailoring for Different Audiences

### Big Tech (Google, Meta, Amazon, etc.)
- Emphasize: scale, systems thinking, impact metrics
- Example: "...systems serving millions of users..." "...reducing latency by 70%..."

### Startups
- Emphasize: versatility, shipping speed, ownership
- Example: "...full-stack capabilities..." "...shipped 3 features in 6 months..."

### Finance/Fintech
- Emphasize: reliability, precision, domain knowledge
- Example: "...trading systems processing 50K+ orders/second..." "...99.99% uptime..."

### Healthcare Tech
- Emphasize: compliance, reliability, domain expertise
- Example: "...HIPAA-compliant systems..." "...healthcare workflow automation..."

## Iteration Process

Writing a good summary usually takes multiple iterations:

1. **First draft**: Get the structure right
2. **Second draft**: Add specific numbers/metrics
3. **Third draft**: Remove generic words
4. **Fourth draft**: Tighten language, reduce words
5. **Final check**: Paste into actual resume format, verify it fits within 2 lines

Don't expect to get it perfect on the first try. Each iteration makes it sharper.

## Length Verification

After writing your summary:
1. Open a Word document or Google Doc
2. Set the font and margins to match your resume format
3. Paste your summary
4. Check if it exceeds 2 lines
5. If too long, identify words to trim:
   - Remove filler adjectives ("various", "multiple", "extensive")
   - Shorten phrases ("in order to" → "to")
   - Consider if every technology needs to be listed
6. Repeat until it fits comfortably
