# The Hiring Manager's Perspective

## How Resumes Really Get Reviewed

### The Reality of Volume

A hot job opening can receive 500-1000+ applications. Recruiters process hundreds of resumes daily. This means:
- Each resume gets an average of 6 seconds of attention
- First impressions determine almost everything
- Recruiters are looking for reasons to say "no," not reasons to say "yes"

### The Two-Pass System

Most companies use a screening process like this:

**First Pass: Recruiter Scan**
- Goal: Filter 500 resumes down to 50
- Time: 6-10 seconds per resume
- Criteria: Does it meet basic requirements (tech stack, years, education)?
- Result: Pass / Fail

**Second Pass: Hiring Manager Review**
- Goal: Filter 50 resumes down to 10-15
- Time: 30-60 seconds per resume
- Criteria: Can this person solve my problem? Any standout qualities?
- Result: Interview / Hold / Pass

The Summary's job is to help you survive the first pass and grab the Hiring Manager's attention in the second pass.

## What Hiring Managers Actually Look For

### In 6 Seconds (Recruiter Scan)
1. Position match: Is this the type of person we're hiring?
2. Tech stack match: Do they have the key technologies?
3. Experience match: Is the years of experience appropriate?
4. Red flag detection: Any obvious problems?

### In 30 Seconds (Hiring Manager Review)
1. Can this person solve my problem?
2. What makes me impressed?
3. Why choose them over others?
4. What would I want to ask in an interview?

## The Mental Model of a Hiring Manager

As a Hiring Manager, I'm hiring because I have a problem to solve:
- The team lacks backend development capability
- Someone needs to build data pipelines
- The system needs ML capabilities
- A project needs someone who can own it

When I look at resumes, I'm asking:
> "Can this person help me solve this problem?"

NOT:
- Is this person impressive
- Does this person know a lot of technologies
- Did this person go to a good school

But rather:
> **Can this person get to work and help me?**

## Calibration Standards

### New Grad Expectations

**Strong New Grad:**
- Clear technical positioning
- 1-2 projects with depth
- Tech stack matches the role
- Shows learning ability and initiative

**Red Flags:**
- Using senior-level vocabulary (Spearheaded, Architected)
- All projects are just homework
- Lists tons of technologies but all surface-level
- Summary direction doesn't match JD

**Acceptable Numbers for New Grads:**
- Users: 100-1000+ users
- Data volume: 10K-100K records
- Performance improvement: 20-50%
- Code coverage: 80%+

**Suspicious Numbers for New Grads:**
- "Serving millions of users" (Where did you get that traffic?)
- "Reduced latency by 90%" (From what to what? How did you measure?)
- "Saved $1M annually" (How would you know that?)

### 1-3 Years Experience Expectations

**Strong 1-3 YoE:**
- Real work impact demonstrated
- Can quantify contributions
- Starting to show ownership
- Technical depth in one or two areas

**Red Flags:**
- Still writing job duties instead of impact
- No numbers at all
- No visible progression from new grad level
- Tech stack too scattered, no focus

**Acceptable Numbers for 1-3 YoE:**
- Users served: 10K-1M users
- Data processed: 1M-100M records
- Performance improvement: 30-70%
- Cost savings: 10-30%

### 3-5 Years Experience Expectations

**Strong 3-5 YoE:**
- Clear ownership and leadership
- Big impact numbers and specific achievements
- Mentorship or team lead experience
- Can drive projects, not just execute

**Red Flags:**
- 5 years experience but resume looks like 2 years
- No leadership indicators
- Impact still at new grad level
- Frequent job hopping with no visible growth

## Common Summary Problems

### Problem 1: Generic Positioning
```
Bad: Software Engineer seeking challenging opportunities to grow.
```
**Why it fails:** What does this tell me? Nothing. Every job seeker can write this sentence.

```
Better: Backend Engineer with 3 years building payment systems in Go and Python.
```
**Why it works:** Specific identity, years, domain, tech stack.

### Problem 2: Technology Dumping
```
Bad: Proficient in Python, Java, Go, JavaScript, TypeScript, C++, Ruby, Scala...
```
**Why it fails:** Do you really know all these languages well? Or did you just learn them? I question your depth.

```
Better: Proficient in Python and Go, with experience in distributed systems and API design.
```
**Why it works:** Focused, shows depth, has application context.

### Problem 3: Vague Claims
```
Bad: Improved system performance significantly.
```
**Why it fails:** What is "significantly"? 50%? 5%? 0.5%?

```
Better: Reduced API latency from 500ms to 150ms through caching and query optimization.
```
**Why it works:** Specific numbers, specific methods.

### Problem 4: Misalignment
```
JD: Data Engineer role focused on building data pipelines
Summary: Full-stack engineer experienced in React and Node.js...
```
**Why it fails:** Why are you applying for this role? Did you apply to the wrong job?

```
Better: Engineer with experience building ETL pipelines and data systems, now seeking to specialize in Data Engineering.
```
**Why it works:** Shows relevant experience, explains the direction.

### Problem 5: Overreach
```
New Grad writing: Spearheaded the architectural transformation of enterprise systems.
```
**Why it fails:** You're a new grad—where did you get the authority to "Spearhead" anything? This word implies leading a team to make strategic changes. Did you do that?

```
Better: Built and deployed a microservices-based application as capstone project.
```
**Why it works:** Realistic, credible, not exaggerated.

## The Comparison Problem

### You're Not Just Competing with the Minimum Bar

When a Hiring Manager looks at your resume, they're NOT asking:
> "Is this person qualified?"

They're asking:
> "Among all the resumes I've seen, where does this person rank?"

Even if you're qualified, if 10 other people are more impressive, you won't get an interview.

### What "Better" Looks Like

Both are New Grads, but the difference between these two Summaries:

**Average:**
> Computer Science graduate with experience in Python and JavaScript. Built several web applications using React. Looking for entry-level software engineering positions.

**Standout:**
> Computer Science new grad who built and shipped a healthcare AI assistant used by 500+ patients. Proficient in Python, LangChain, and AWS. Seeking Backend/AI roles in health tech.

What's the difference?
1. Specific project (not "several web applications")
2. Real user numbers (500+ patients)
3. Vertical positioning (health tech)
4. Tech stack and direction alignment

## Scoring Rubric

### Position Fit - 25%
- 10/10: Perfect match with JD's core requirements
- 7-9: Generally matches, some minor gaps
- 4-6: Partial match, needs explanation
- 1-3: Clear mismatch, looks like wrong application

### Credibility - 25%
- 10/10: All claims have concrete evidence
- 7-9: Mostly credible, a few need verification
- 4-6: Some exaggeration or vagueness
- 1-3: Obviously not credible or has red flags

### Differentiation - 20%
- 10/10: Has unique highlights that are memorable
- 7-9: Has some differentiating points
- 4-6: Nothing special
- 1-3: Same as 500 other resumes

### Scannability - 15%
- 10/10: Can understand core info in 3 seconds
- 7-9: Takes 5-10 seconds
- 4-6: Requires careful reading
- 1-3: Confusing, can't find the key points

### Language Quality - 15%
- 10/10: Professional, fluent, error-free
- 7-9: Generally professional, minor issues
- 4-6: Some unprofessional elements or errors
- 1-3: Grammar errors or confusing expression

## The Rewrite Test

A good review doesn't just point out problems—it shows how to improve.

**Before:**
> Passionate software developer with experience in various technologies seeking challenging opportunities in a dynamic environment.

**After:**
> Backend Engineer with 2 years building REST APIs in Python and Go. Reduced p99 latency by 40% through query optimization at [Company]. Seeking senior backend roles at growth-stage startups.

What changed?
- "Passionate" → Deleted, useless
- "various technologies" → Specify what technologies
- "challenging opportunities" → State what role you want
- Added specific achievements and numbers
- Added target company type
